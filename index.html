Dredd

No more outdated API documentation

Download this project as a .zip file Download this project as a tar.gz file

Dredd is a command-line tool for validating API documentation written in API Blueprint format against its backend implementation built on top of Gavel HTTP validation library.

Installation

Node.js and NPM is required.

$ npm install -g dredd

Get Started

$ cat > apiary.apib <<EOF
# GET /message
+ Response 200 (text/plain)

      Hello World!
EOF
$ dredd blueprint.md http://localhost:3000
info: Beginning Dredd testing...
pass: GET /message duration: 9ms
complete: 1 passing, 0 failing, 0 errors, 0 skipped
complete: Tests took 16ms

Continous Integration

With Dredd you can easily plug your API documentation into the Continous Integration system like Travis CI or Jenkins and have API documentation up-to-date, all the time.

Validating real-life API

See dredd-example repo for example application.

In previous post we introduced Dredd and the tutorial how you can test your API blueprints with Dredd and make them a part of your project's testing harness in the continous integration. The benefit of this approach is always valid API documentation. I'd like to show here more practical example how to test an already existing API documentation and which catches you can face to.

Examples for URI Parameters

First of all make sure that resources specified by an URI template in the blueprint have specified an example value for each URI parameter in resource's parameters section:

## Gist [/gists/{id}]

+ Parameters
    + id (required,`42`) ... ID of the Gist in the form of a hash.

Resources defined with URI template without specified parameter example value are considered as ambigous. You can check it simply by running dredd --names apiary.apib

Resource lifecycle

Not all bleuprints are written as scenario oriented. It means that order of resources declaration in the blueprint shall not respect entity's lifecycle. For example retreiving a collection of Gists before creating it.

If your API maintain complete entity lifecycle under one single URI endpoint, you can use --sort switch and Dredd will automatically sort resource actions starting with POST and ending with DELETE. Otherwise if the entity lifecycle is split into more URI endpoints (collections and members for example) you have to use Hooks and fixtures for each action to setup and teardown testable situation.
