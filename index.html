Dredd is a command-line tool for validating API documentation written in API Blueprint format against its backend implementation built on top of Gavel HTTP validation library.

Installation

Node.js and NPM is required.

$ npm install -g dredd

Get Started

$ cat > apiary.apib <<EOF
# GET /message
+ Response 200 (text/plain)

      Hello World!
EOF
$ dredd blueprint.md http://localhost:3000
info: Beginning Dredd testing...
pass: GET /message duration: 9ms
complete: 1 passing, 0 failing, 0 errors, 0 skipped
complete: Tests took 16ms

Continous Integration

With Dredd you can easily plug your API documentation into the Continous Integration system like Travis CI or Jenkins and have API documentation up-to-date, all the time.

Validating real-life API Blueprint

Let's try more complicated example. Here you can find complete documentation-driven application in Express.js for the GistFox API used in the Apiary API Blueprint tutorial.

When you want to validate real-life API blueprint you will face two main catches:

  1. Missing URI parameter example values
  2. Non testable resource entity lifecycle

1. Examples for URI Parameters

First of all make sure that all resources specified by an URI template have specified an example value for each URI parameter in the resource's parameters section:

Exapmle:

## Gist [/gists/{id}]

+ Parameters
    + id (required,`42`) ... ID of the Gist in the form of a hash.

Resources defined with URI template without specified parameter example value are considered as ambigous. You can check it simply by running dredd --names apiary.apib http://localhost:3000

2. Testable entity lifecycle

Not all bleuprints are written as scenario oriented. It means that order of resources declared in the blueprint shall not respect entity's lifecycle. For example retreiving a collection of Gists before creating it.

Workaround for sorting actions

If your API maintain complete entity's lifecycle under one single URI endpoint, you can use --sort switch to make Dredd automatically sort actions for the resource starting with POST and ending with DELETE.

Using hook files and fixtures

Dredd allows writing before and after hooks for each defined actions in JavaScript or CoffeeScript.

If the entity lifecycle is split into more URI endpoints (e.g. collections and members) the solution is to use Dredd Hooks and database fixtures to setup and teardown testable situation.

When you have a look at the GistFox API documentation or source used in the example-application you can see that API actions on a single Gist entity are defined before action for Gist creation (adding to the collection).

To setup a testable situation for showing a Gist, you have to create it by adding a fixture to the dabase in the before hook:

  1. Find name of action which you want to wrap with a hook:

    $ dredd ./apiary.apib http://localhost:3000 --names | grep Retreive
info: Gist > Gist > Retrieve a Single Gist

  2. Create a hook file in hooks/hookfile.coffee:

    {before, after} = require 'hooks'

db = require("mongous").Mongous
gists = db('test.gists')
gist =
  "id": "42",
  "created_at": "2014-04-14T02:15:15Z"
  "description": "Description of Gist"
  "content": "String contents"

before "Gist > Gist > Retrieve a Single Gist", (transaction) ->
  gists.insert gist

  3. Let Dredd know about hooks

    $ dredd apiary.apib http://localhost:3000/ --hookfiles=hooks/*
info: Found Hookfiles: hooks/hooks.coffee
